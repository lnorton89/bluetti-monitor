<objective>
Extract shared UI surface components that replace the route-specific card, panel, header, metric, chip, and row patterns that currently drift across Overview, Charts, Solar, and RawData pages.

Purpose: The four dashboard pages each define their own card headers, metric displays, status chips, info rows, section panels, and empty states using different CSS classes and inline structures. The result is visual drift, redundant code, and inconsistent spacing/typography. This plan creates reusable components and CSS tokens that all pages will share.

Output: 7 new shared components + refactored CSS token classes + barrel exports from ui.tsx.
</objective>

  <name>Task 1: Extract shared CSS token classes from duplicated route patterns</name>
  <files>dashboard/src/index.css</files>
  <action>
Analyze the current `index.css` and extract the duplicated visual patterns into shared CSS token classes. The goal is to eliminate the route-specific CSS classes that all do the same thing with different names, replacing them with unified token classes.

**Identify and collapse these duplicated patterns:**

1. **Card surfaces** — `.summary-card`, `.info-panel`, `.category-panel`, `.workspace-panel`, `.analytics-hero-card`, `.analytics-report-card`, `.analytics-insight-card`, `.analytics-score-card`, `.analytics-explorer-card`, `.solar-input-detail-card`, `.solar-mapping-card`, `.data-table-shell`, `.status-board-card`, `.ac500-note-card` all share: flex column, gap ~14-18px, background gradient. Create a `.surface-card` token class for generic card surfaces with gradient backgrounds, and keep page-specific gradient accents as modifier classes (e.g., `.surface-card--accent-warm`, `.surface-card--accent-ink`, `.surface-card--accent-solar`).

2. **Section headers** — `.info-panel-title`, `.workspace-panel-title`, `.workspace-panel-kicker`, `.category-panel-title`, `.analytics-hero-title`, `.analytics-report-title`, `.chart-card-title`, `.solar-input-card-header span` all follow: condensed uppercase font, icon + label, kicker text above. Create `.section-title`, `.section-kicker`, `.section-header` token classes.

3. **Metric displays** — `.stat-tile`, `.summary-card-value + .summary-card-label`, `.analytics-score-value + .analytics-score-label`, `.solar-input-card-header strong + span`, `.category-highlight-chip span + strong` all follow: label (dim, condensed) + value (mono, large) + optional detail. Create `.metric-tile`, `.metric-label`, `.metric-value`, `.metric-detail` token classes.

4. **Chips/pills** — `.mode-chip`, `.data-chip`, `.stale-indicator`, `.device-status-pill`, `.workspace-pill`, `.analytics-segment-button`, `.analytics-selection-chip`, `.category-highlight-chip`, `.top-bar-metric` all follow: inline-flex, pill radius, border, condensed font. Create `.chip`, `.chip--active`, `.chip--status` token classes.

5. **Info rows** — `.info-row`, `.switchboard-item`, `.solar-input-card-row`, `.solar-mapping-row`, `.analytics-field-row`, `.power-node-split` all follow: flex row, space-between, label + value. Create `.info-row` (already exists, formalize), `.info-row-label`, `.info-row-value`.

6. **Subtle grids** — `.stat-grid`, `.summary-grid`, `.switchboard-grid`, `.analytics-score-grid`, `.solar-score-grid`, `.solar-insights-grid` all follow: `auto-fit, minmax(Xpx, 1fr)` or `repeat(N, 1fr)` grids. Create `.tile-grid` and `.tile-grid--cols-2` through `.tile-grid--cols-4` token classes.

7. **Page-level containers** — `.overview-page`, `.page-stack`, `.device-section` all follow: flex column with consistent gap. Keep these as they are (they're page-specific layout concerns, not draggable cards).

**Implementation approach:**
- Add token classes in a clearly labeled section `/* ── Shared surface tokens ── */` near the top of the layout styles (before `.card`)
- Keep the existing route-specific classes as aliases pointing to token classes using CSS comma selectors: `.summary-card, .info-panel, .surface-card { ... }` 
- This avoids breaking any existing pages until Plan 02 and Plan 03 refactor them
- Remove inline `style` objects on shared components (Card, etc.) where CSS token classes can replace them
- All token classes should use CSS custom properties already defined in `:root` (e.g., `var(--bg-2)`, `var(--font-cond)`, `var(--border)`, etc.)
